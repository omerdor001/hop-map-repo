A full-stack child safety platform that detects and alerts parents when children attempt to "hop" from moderated gaming environments to unmoderated external platforms in real time.

## üéØ Overview

HopMap monitors a child's Windows gaming session and uses LLM-powered classification to detect **platform hopping** ‚Äî when a child is lured from a supervised game to an unmoderated platform (Discord, Instagram, Telegram, etc.). The system includes:

- **Desktop Agent** ‚Äî Lightweight Windows sensor using Win32 hooks, Tesseract OCR, and clipboard monitoring with no local LLM overhead
- **Agent Installer** ‚Äî Self-contained PowerShell installer generated by the server; parents download a ZIP from the dashboard and run it once on the child's PC ‚Äî it installs Python, Tesseract, the agent, and registers a Windows Task Scheduler task automatically
- **Classification Server** ‚Äî FastAPI backend that supports Ollama (local) or NVIDIA NIM (cloud) for URL/context classification, keeping all AI inference off the child's machine
- **Blocked-Words Filter** ‚Äî Aho-Corasick multi-pattern filter that intercepts known harmful words and phrases before any LLM call, supporting special characters (`18+`) and Unicode scripts (Hebrew)
- **Parent Auth** ‚Äî JWT access tokens (15 min) + httpOnly refresh token cookies (30 days) with bcrypt password hashing, SHA-256 agent token storage, server-side password complexity validation, per-IP rate limiting on auth endpoints, and email-based password reset (SMTP, enumeration-safe)
- **Parent Dashboard** ‚Äî React 19 frontend with live SSE event streaming, child profile management (add/delete), alert history, subscriptions, one-click installer download, whitelist/blacklist management, and a "Forgot password?" flow
- **MongoDB Database** ‚Äî Persistent storage for hop events, children, auth sessions, notifications, and blocked words
- **Health Checks** ‚Äî `/health/live` and `/health/ready` endpoints reporting MongoDB, Ollama, and words-filter status
- **Test Suite** ‚Äî 736 passing tests across unit, integration, and E2E layers

## üöÄ Quick Start

### Prerequisites

```bash
# Python 3.10+, Node.js 18+, MongoDB running locally or on Atlas
# React 19 (installed automatically via npm install)

# Option A ‚Äî Local inference (Ollama):
# Install Ollama desktop client from https://ollama.com and run:
ollama pull qwen2.5:7b

# Option B ‚Äî Cloud inference (NVIDIA NIM):
# Get an API key from https://build.nvidia.com and set it in server/.env:
# HOPMAP_SERVER__LLM__API_KEY=nvapi-...
# Then set provider + model in server/server_config.json (see Configuration)
```

### Setup & Run

1. **Configure the server** ‚Äî edit `server/server_config.json` for operational settings, then copy `server/.env.example` to `server/.env` and fill in secrets:
   ```
   HOPMAP_SERVER__DB__MONGO_URI=mongodb://localhost:27017
   HOPMAP_SERVER__AUTH__JWT_SECRET=your-secret-here
   HOPMAP_SERVER__LLM__API_KEY=nvapi-...   # NVIDIA NIM only

   # Optional ‚Äî enables the password reset email flow
   HOPMAP_SERVER__EMAIL__ENABLED=true
   HOPMAP_SERVER__EMAIL__SMTP_USER=you@gmail.com
   HOPMAP_SERVER__EMAIL__SMTP_PASSWORD=app-password-here
   HOPMAP_SERVER__EMAIL__RESET_PASSWORD_URL=http://localhost:5173
   ```

2. **Start the server**:
   ```bash
   cd server
   pip install -r requirements.txt
   uvicorn server:app --host 0.0.0.0 --port 8000
   ```

3. **Start the dashboard** (in a new terminal):
   ```bash
   cd dashboard
   npm install
   npm run dev
   ```

4. **Open the parent dashboard**:
   Navigate to `http://localhost:5173`, register an account, and add a child profile.

5. **Install the agent on the child's PC**:
   - In the dashboard go to **Kids ‚Üí Add Child**, enter the child's name, then click **Download Installer**
   - Copy the ZIP to the child's Windows PC
   - Right-click `hopmap_install.ps1` ‚Üí **Run with PowerShell** and click Yes on the UAC prompt
   - The script installs Python, Tesseract OCR, downloads the agent files, activates the agent token, and registers a Task Scheduler task that starts the agent on every login automatically

### Component Flow

1. **Agent** detects a candidate URL via OCR screenshot, clipboard poll, or window title
2. **Agent ‚Üí Server** sends context snippet to `POST /agent/classify`; server runs the blocked-words filter first (O(n) Aho-Corasick pass), then the local Ollama model if no match, and returns `{ decision, confidence, reason, via }`
3. **Agent** observes the subsequent app-switch and confirms the hop tier (`app_match`, `title_match`, or `switch_only`)
4. **Agent ‚Üí Server** reports the confirmed hop via `POST /agent/hop/{child_id}`; server persists the event to MongoDB and creates a parent notification
5. **Server ‚Üí Dashboard** pushes the event over SSE to all connected parent browsers watching that child

## üîß Components

### 1. Desktop Agent (`agent/`)

A lightweight Windows-only sensor that runs on the child's PC. All LLM classification is delegated to the server, keeping gaming performance unaffected.

**Detection Pipeline:**

| Layer | Mechanism | Description |
|-------|-----------|-------------|
| Layer 1 | Win32 `SetWinEventHook` | Fires on every foreground-window change; HWNDs queued immediately |
| Layer 2 | Event-processor thread | Drains the HWND queue; drives scanner lifecycle and hop confirmation |
| Layer 3 | OCR scanner (Tesseract + mss) | Screenshots active game window every ~5 sec; extracts URL candidates |
| Layer 4 | Clipboard monitor | Polls clipboard every ~1 sec ‚Äî catches "copy this link" lures invisible to OCR |
| Layer 5 | Classification & confirmation | TTL-deduplicates URLs, POSTs to `/agent/classify`, confirms hop tier on next app-switch |

**Confirmation Tiers:**

| Tier | Signal |
|------|--------|
| `app_match` | The native desktop app for the lure platform opened |
| `title_match` | A browser navigated to the lure domain (title poll) |
| `switch_only` | An app switch occurred but no stronger signal was available |

**Platform defaults (used if server unreachable):** Discord, Telegram, WhatsApp, Signal, Snapchat, Instagram, TikTok, YouTube, Twitch ‚Äî plus common browsers (Chrome, Edge, Firefox, Opera, Brave, Vivaldi).

### 2. Classification Server (`server/`)

A FastAPI application that runs on the parent's network. Provides endpoints for the agent, real-time SSE for the dashboard, and REST management APIs.

**Classification Flow:**

1. Context snippet arrives at `POST /agent/classify`
2. `check_blocked_words()` runs an O(n) Aho-Corasick scan ‚Äî supports phrases, special chars, Unicode
3. If a blocked word/phrase matches ‚Üí immediately return `{ decision: YES, via: word_db }` (no LLM call)
4. Otherwise ‚Üí forward to the configured LLM provider (Ollama or NVIDIA NIM); return `{ decision, confidence, reason, via: server }`

### 4. Agent Installer (`server/agent_installer/`)

The parent never manually configures the agent. Instead:

1. Dashboard calls `GET /agent/installer?childId=‚Ä¶&backendUrl=‚Ä¶`
2. Server generates a self-contained PowerShell script with a one-time `setup_code` embedded, packages it with an uninstaller and README into a `.zip`
3. Parent copies the ZIP to the child's PC and runs `hopmap_install.ps1` as Administrator
4. The script installs Python, Tesseract OCR, downloads the latest agent files, writes `agent_config.json` (UTF-8, no BOM), registers a Task Scheduler task, and starts it immediately
5. On first run the agent calls `POST /agent/activate` to exchange the setup code for a permanent `agentToken`, then clears the setup code from `agent_config.json`

## üõ°Ô∏è Security Notice

- **Pluggable LLM Backend** ‚Äî Supports Ollama (fully local, no data leaves the network) or NVIDIA NIM (cloud API); choose based on your privacy and performance requirements
- **Offline-capable** ‚Äî With Ollama, works fully offline on a local network (no cloud dependency)
- **Minimal Agent Footprint** ‚Äî The desktop agent sends only small context snippets (a few lines of chat + detected URL), not full screen captures
- **JWT + Refresh Tokens** ‚Äî Short-lived access tokens (15 min) with httpOnly refresh cookies; SHA-256 hashing prevents rainbow table attacks on stored tokens
- **Password Complexity** ‚Äî Server-side Pydantic validator enforces min-length 8, uppercase, lowercase, digit, and special character; frontend shows a real-time zxcvbn strength meter and blocks submission below "Fair" (score ‚â• 2)
- **Password Reset** ‚Äî Email-based reset flow: 256-bit URL-safe token (`secrets.token_urlsafe(32)`), 30-minute expiry, one-time-use (atomically cleared on redemption); `POST /auth/forgot-password` always returns 200 regardless of whether the email exists, preventing enumeration
- **Auth Rate Limiting** ‚Äî Per-IP rate limits on `/auth/login` (10/min), `/auth/register` (5/min), and `/auth/forgot-password` (5/min) via slowapi; returns 429 with `Retry-After` header (RFC 6585)
- **Classify Rate Limiting** ‚Äî Per-child async rate limiter (30 RPM default) enforced on the classify endpoint
- **MongoDB Circuit Breaker** ‚Äî Three-state circuit breaker (CLOSED ‚Üí OPEN ‚Üí HALF_OPEN) suppresses DB calls after 5 consecutive failures and returns HTTP 503 with `Retry-After` until the circuit self-heals (30 s default)
- **Startup Secret Validation** ‚Äî Server checks `JWT_SECRET` length (‚â• 32 chars) and rejects the well-known default value; violations are fatal in `production`/`staging` (process exits with code 1) and logged as warnings in `development`
- **Graceful Degradation** ‚Äî If the server is unreachable, the agent skips classification and treats the URL as safe rather than blocking the child's session
- **Agent Token Isolation** ‚Äî After activation the agent token is removed from `agent_config.json` and stored exclusively in Windows Credential Manager (via `keyring`); the JSON config file never contains a valid token at rest
- **Hop Delivery Reliability** ‚Äî Hop events are retried with exponential backoff (5 s ‚Üí 10 s ‚Üí 20 s ‚Üí 30 s) and silently dropped only after all attempts fail or the event exceeds a 5-minute TTL; 4xx rejections are dropped immediately without retrying
